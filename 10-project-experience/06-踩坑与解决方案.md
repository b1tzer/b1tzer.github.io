# 踩坑与解决方案

---

## 坑1：浏览量统计不准确（Redis 计数丢失）

### 现象
服务重启后，部分问题的浏览量回退到几小时前的数值。

### 原因
浏览量增量存在 Redis 中，定时任务每 5 分钟将增量持久化到 MySQL。服务重启时，Redis 中尚未持久化的增量数据丢失。

### 解决方案
1. **Redis 开启 AOF 持久化**：减少数据丢失窗口
2. **定时任务缩短间隔**：从 5 分钟改为 1 分钟
3. **服务关闭时主动刷盘**：监听 `ContextClosedEvent`，在服务停止前将 Redis 中的计数全部持久化到 MySQL

```java
@EventListener(ContextClosedEvent.class)
public void onShutdown() {
    log.info("服务关闭，开始持久化浏览量数据...");
    viewCountService.flushAllToDatabase();
}
```

## 坑2：点赞数出现负数

### 现象
数据库中某些问题的 `like_count` 出现了负数。

### 原因
用户取消点赞时，直接执行 `UPDATE question SET like_count = like_count - 1`，但没有校验当前值是否大于 0。在极端情况下（数据不一致时），会导致负数。

### 解决方案
```sql
-- 加上 > 0 的条件保护
UPDATE question SET like_count = like_count - 1 
WHERE id = ? AND like_count > 0;
```

同时在 Redis 层也加保护：
```java
Long result = redisTemplate.opsForValue()
    .decrement("question:like:count:" + questionId);
if (result != null && result < 0) {
    // 修正为 0
    redisTemplate.opsForValue().set("question:like:count:" + questionId, 0);
}
```

## 坑3：ES 搜索结果包含已删除的问题

### 现象
问题被删除后，在搜索结果中仍然能搜到。

### 原因
删除问题时，Kafka 消息发送失败（网络抖动），搜索服务没有收到删除事件，ES 中的文档未被删除。

### 解决方案
1. **Kafka 生产者开启重试**：`retries: 3`
2. **本地消息表兜底**：业务操作和消息记录同事务，独立投递服务保证消息最终发出
3. **定期对账**：每天凌晨扫描 MySQL 中已删除的问题 ID，与 ES 对比，删除 ES 中多余的文档

```java
// 对账任务
@Scheduled(cron = "0 0 2 * * ?")
public void reconcile() {
    // 查询 MySQL 中最近7天删除的问题ID
    List<Long> deletedIds = questionMapper.selectRecentDeletedIds(7);
    for (Long id : deletedIds) {
        // 检查 ES 中是否还存在
        if (esClient.exists(id)) {
            esClient.delete(id);
            log.warn("对账删除 ES 文档: questionId={}", id);
        }
    }
}
```

## 坑4：圈子热门问题 Redis Key 过多

### 现象
随着圈子数量增加，Redis 中 `hot:questions:circle:*` 的 Key 越来越多，内存占用持续增长。

### 原因
每个圈子都有一个独立的 ZSet，存储该圈子的热门问题列表，且没有设置过期时间。

### 解决方案
1. **设置过期时间**：热门榜单 Key 设置 7 天过期，定时任务每天重新计算并刷新
2. **限制 ZSet 大小**：每个圈子只保留 Top 100 的问题，超出时删除分数最低的

```java
// 更新热度分后，裁剪 ZSet 只保留 Top 100
redisTemplate.opsForZSet().add("hot:questions:circle:" + circleId, questionId, score);
redisTemplate.opsForZSet().removeRange("hot:questions:circle:" + circleId, 0, -101);
// 刷新过期时间
redisTemplate.expire("hot:questions:circle:" + circleId, Duration.ofDays(7));
```

## 坑5：@用户通知重复发送

### 现象
用户编辑问题时，如果内容中有 @某人，该用户会收到重复的通知。

### 原因
每次编辑保存都会重新解析 @ 列表并发送通知，没有区分"新增的 @"和"已有的 @"。

### 解决方案
编辑时对比新旧内容中的 @ 用户列表，只对**新增的 @** 发送通知：

```java
public void updateQuestion(Long questionId, String newContent) {
    Question old = questionMapper.selectById(questionId);
    Set<Long> oldMentions = parseMentions(old.getContent());
    Set<Long> newMentions = parseMentions(newContent);
    
    // 只有新增的 @ 才发通知
    Set<Long> addedMentions = new HashSet<>(newMentions);
    addedMentions.removeAll(oldMentions);
    
    if (!addedMentions.isEmpty()) {
        kafkaTemplate.send("mention-events", new MentionEvent(questionId, addedMentions));
    }
}
```

## 坑6：微服务间调用超时导致雪崩

### 现象
ES 服务响应变慢时，搜索服务的线程池被打满，进而导致网关也无法响应，整个系统不可用。

### 原因
没有配置服务间调用的超时和熔断，一个服务慢导致调用方线程全部阻塞。

### 解决方案
1. **配置 Feign 超时**：连接超时 1s，读取超时 3s
2. **引入 Sentinel 熔断降级**：ES 服务不可用时，搜索接口返回降级结果（提示"搜索服务暂时不可用，请稍后重试"）
3. **线程池隔离**：搜索服务使用独立线程池，不影响其他接口

```yaml
feign:
  client:
    config:
      search-service:
        connect-timeout: 1000
        read-timeout: 3000
```

```java
@FeignClient(name = "search-service", fallback = SearchServiceFallback.class)
public interface SearchServiceClient {
    @GetMapping("/search")
    Result<Page<QuestionVO>> search(@RequestParam String keyword);
}

@Component
public class SearchServiceFallback implements SearchServiceClient {
    @Override
    public Result<Page<QuestionVO>> search(String keyword) {
        return Result.fail("搜索服务暂时不可用，请稍后重试");
    }
}
```

## 总结

| 坑 | 根本原因 | 核心解法 |
|----|---------|---------|
| 浏览量丢失 | Redis 数据未持久化 | 优雅关闭 + AOF + 缩短刷盘间隔 |
| 点赞数负数 | 缺少边界校验 | SQL 加 `> 0` 条件 |
| ES 数据不一致 | 消息丢失 | 本地消息表 + 定期对账 |
| Redis 内存膨胀 | Key 无过期 + 无大小限制 | 设置 TTL + ZSet 裁剪 |
| 通知重复发送 | 未做差量计算 | 新旧内容 diff |
| 服务雪崩 | 无超时无熔断 | Feign 超时 + Sentinel 熔断 |
